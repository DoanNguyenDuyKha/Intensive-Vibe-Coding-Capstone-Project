## 1. The Problem: Static Dashboards in a Dynamic World

Every business analyst has experienced the same frustration: you need to answer a specific question — "Who was the top-performing sales rep in Q4?" — but your dashboard only shows last month's total revenue. Getting the answer requires submitting a ticket to the data team, waiting days for a developer to modify SQL queries, update the dashboard configuration, and redeploy. By the time the answer arrives, the business decision has already been made — or missed.

This problem exists across industries. Business intelligence tools like Tableau, Power BI, and Looker are powerful but static: they show what developers pre-configured, not what analysts actually need right now. Analysts either accept this limitation or export raw CSVs and struggle with Excel — neither of which scales.

**A2UI Data Canvas** is my answer to this problem. It is an AI agent-powered sales intelligence dashboard that lets any business user ask any question in plain language and receive a fully rendered, interactive, multi-card dashboard in seconds. No SQL knowledge required. No developer ticket. No waiting.

The system is designed for the **Agents for Business** track because it directly addresses an enterprise pain point — data accessibility — through meaningful, accountable AI agent design.

## 2. Why Agents? The Design Rationale

Before building, I asked myself: why use agents instead of a single prompt?

A single LLM prompt cannot reliably handle the full pipeline:

1. **Intent disambiguation** — Is the user asking to *see* data or *change* data?
2. **Query generation** — Translating vague NL to precise, safe SQL requires domain knowledge and schema awareness
3. **Data retrieval** — Fetching from a real database requires tool calls, not just LLM output
4. **UI layout design** — Selecting the right chart type, card structure, and visual hierarchy for the specific data shape
5. **Schema validation** — Ensuring the generated JSON strictly adheres to the A2UI v0.9 specification
6. **Self-correction** — Detecting validation failures and retrying with structured error context

Bundling all of this into one prompt produces brittle, unpredictable results. An **ADK Graph Workflow** with specialized nodes handles each step accountably, enables self-correction loops, and integrates MCP tools for real database access.

## 3. System Architecture

### The Four-Node ADK Graph Workflow

The core of the system is a `sales_canvas_workflow` built using **Google ADK 2.0's Graph Workflow** API:

```
START
  → Intent_Router
  → Data_Fetcher
  → UI_Generator_Agent
  → Critic_Node
  → Final Output
```

**Intent_Router** is the entry point. It analyzes the user prompt to determine whether this is a data retrieval request (SELECT) or a data modification request (UPDATE). For modification requests, it uses Gemini to translate the natural language instruction into a valid SQL UPDATE statement and executes it through the MCP `update_sales_data` tool before proceeding. This separation prevents a common failure mode where a single-purpose data agent confuses read and write operations.

**Data_Fetcher** is the intelligence hub. It receives the user prompt and constructs a precise SQL SELECT query using Gemini with an extensive system prompt containing 11 specific aggregation patterns. This is the most important design decision in the system: by explicitly teaching the SQL generator patterns for ranking queries, time-series trends, proportional shares, and conditional aggregations (Q3 vs Q4 comparisons), I eliminated the hallucination of non-existent column names and incorrect GROUP BY structures that plagued early iterations. The query is executed through the MCP `query_sales_data` read-only tool.

**UI_Generator_Agent** takes the raw query results and designs a complete multi-card dashboard. It uses Gemini with schema-constrained JSON output (`response_mime_type="application/json"`) and the A2UI v0.9 system prompt generated by the `A2uiSchemaManager`. The output must be a valid "Hybrid Output" containing both the raw data (for chart rendering on the client) and the A2UI component tree. Critically, this node implements a **3-attempt self-correction loop**: if the generated JSON fails schema validation, the exact validation error is injected into the next prompt, allowing the model to correct its own mistakes without human intervention.

**Critic_Node** is the final circuit breaker. It performs a lightweight pass-through validation to confirm that the output from the UI Generator is complete and valid before returning it to the client. This prevents partial or corrupted outputs from reaching the browser.

### MCP SQLite Server

The **Model Context Protocol server** (`mcp_server.py`) is built with FastMCP using stdio transport. It exposes two isolated tools:

- `query_sales_data(sql: str)` — executes only SELECT statements; any non-SELECT input raises a validation error
- `update_sales_data(sql: str)` — executes only UPDATE/INSERT statements; any SELECT attempt is blocked

This permission isolation is a critical security design: different workflow nodes have access to different tools based on their role, preventing the Data_Fetcher from accidentally destroying data.

### A2UI Protocol Integration

The frontend renders dashboards using the **A2UI v0.9 Basic Catalog** — a declarative JSON protocol for describing UI components. The AI generates a component tree (Column, Row, Card, Text, Button) that the browser's A2UI renderer translates into live HTML. This decouples dashboard design from HTML/CSS knowledge: the AI only needs to understand the component schema, not browser rendering.

## 4. Course Concepts Applied

This project applies five core concepts from the Intensive Vibe Coding course:

**Multi-Agent Graph Workflow:** The four-node ADK workflow graph is the backbone of the system. Each node has a single responsibility, shared state flows through a typed `SalesDashboardState` Pydantic model, and edges define the execution order. The graph is registered as an ADK `App` and served via FastAPI.

**Model Context Protocol:** The FastMCP server exposes real SQLite database access to the agents. Using MCP rather than direct database calls means the agents interact with a well-defined tool interface — one that can be swapped for a BigQuery or Cloud SQL backend without changing agent code.

**Agent-to-UI Protocol (A2UI v0.9):** The entire frontend rendering pipeline is driven by AI-generated A2UI JSON. The `A2uiSchemaManager` generates the system prompt for the UI Generator, and the `A2uiValidator` validates every output before delivery. This is the novel part of the project: the AI does not write HTML; it writes structured component trees that are rendered client-side.

**Self-Correction Reflection:** The 3-attempt retry loop in `UI_Generator_Agent` is a practical application of the reflection pattern. Instead of crashing on invalid output, the system feeds the exact jsonschema validation error back to the model as context. In testing, this reduced invalid-schema failures from ~40% on first attempt to near-zero on final delivery.

**Security and Guardrails:** MCP tool-level permission isolation, SQL injection prevention via allow-list validation, gitignored credentials, and schema-enforced output all implement defense-in-depth security appropriate for enterprise deployments.

## 5. Technical Implementation Highlights

### Model Fallback Chain

To handle free-tier API quota exhaustion during development and demos, I implemented an automatic model fallback chain: `gemini-2.5-flash → gemini-3.5-flash → gemini-3.1-flash-lite → gemini-flash-latest`. This ensures the demo never fails due to daily quota limits.

### SQL Generation Quality

The SQL generation system prompt contains 11 specific pattern categories (ranking, comparison, time-series, filtering, proportional, average, multi-metric, Q3-vs-Q4, detailed profile, etc.) with concrete examples for each. This in-prompt few-shot learning dramatically improved query accuracy over a generic "generate SQL" instruction.

### Frontend Chart Engine

The frontend implements four chart types entirely in vanilla JavaScript without external chart libraries:
- **Bar chart** with spring-physics height animation and SVG solid fills (print-safe)
- **Line chart** with SVG area fill, gradient stroke, and clip-path sweep animation
- **Donut chart** with staggered stroke-dasharray animation (segments bloom one by one)
- **Inline mini-donut** rendered inside Card components for distribution data

### A2UI Windows Compatibility Fix

A non-trivial challenge was that the A2UI jsonschema validator uses `os.path.join` for JSON Schema `$ref` resolution, which produces Windows backslashes in file URIs that break the resolver. I implemented a monkeypatch that temporarily replaces `os.path.join` and `os.path.dirname` with their `posixpath` equivalents during validator construction. This is documented as a comment in `agent.py`.

## 6. Project Journey

The project evolved through distinct phases:

**Phase 1 — Architecture Design:** Started with a single-agent approach and quickly found that combining SQL generation, data fetching, UI design, and validation in one prompt produced inconsistent results. Refactored to the 4-node graph workflow.

**Phase 2 — MCP Integration:** Building the FastMCP server and integrating it with the ADK workflow required careful handling of the stdio transport in the Windows environment. The key insight was importing MCP tool functions directly into agent.py rather than using network-based MCP calls, which simplified the local development loop.

**Phase 3 — UI Quality:** Early A2UI outputs were technically valid but visually poor — single cards with plain text. I invested significant effort in the system prompt's Layout Intelligence Rules, teaching the model to always generate multi-card dashboards with title cards, executive summary cards, KPI metric grids, and breakdown cards with animated progress bars.

**Phase 4 — Animation and Polish:** The frontend chart engine went through three major revisions. The final version uses CSS animation keyframes with spring-physics cubic-bezier timing (`cubic-bezier(0.34, 1.56, 0.64, 1)`) for a premium feel. Print support required switching from CSS gradients (stripped by browser print engines) to inline SVG with solid fills.

**Phase 5 — Testing:** Built a 7-test suite covering offline unit tests, live integration tests, and full E2E SSE stream tests. The offline tests use mock patches to run without API keys, enabling fast CI feedback.

The completed system delivers:
- **Sub-10 second** response time from prompt submission to full dashboard render
- **7/7 tests passing** across unit, integration, and E2E test suites
- Support for **11 distinct query patterns** including ranking, time-series, comparison, and cross-quarter analysis
- **Self-correcting output** with near-zero schema validation failures reaching the client
- A dashboard that **any business user** can operate without SQL or developer knowledge
